## 1. 引言

### 1.1 项目背景
软件体系结构风格的选择直接影响系统的质量属性（性能、可扩展性、可维护性等）。传统上，架构选型依赖架构师个人经验和手工分析，面对日益复杂的业务需求和非功能性约束，迫切需要智能化辅助工具。本系统利用大语言模型（LLM）的语义理解能力和多智能体协作机制，为软件需求自动推荐最合适的体系结构风格。

### 1.2 项目目标
设计并实现一个软件体系结构风格智能助手，结合大语言模型、智能体（Agent）和微服务技术，实现需求分析、架构推荐、决策支持与知识进化四大核心功能。

## 2. 系统概述

### 2.1 产品范围
本系统面向软件架构师、技术负责人及开发者，提供从自然语言需求描述到架构推荐的全流程辅助。系统以 Web 应用形式交付，支持浏览器交互和 API 接口调用。

### 2.2 用户角色
| 角色 | 描述 |
|------|------|
| 普通用户 | 输入需求描述，获取架构推荐与评估报告 |
| 知识管理员 | 维护架构知识库，添加新架构风格与案例 |

### 2.3 核心功能
1. **需求理解** — 接收自然语言需求描述，提取关键特征（并发量、一致性要求、扩展需求等）
2. **架构推荐** — 匹配≥3种候选体系结构风格，按匹配度排序
3. **决策支持** — 多维度对比分析（可扩展性/性能/耦合度/复杂度/部署灵活/可测试性）、需求定制拓扑图、决策溯源和组合推荐
4. **知识进化** — 自动保存分析案例，当前案例库包含 36 条历史记录，利用 Few-shot 学习持续优化推荐质量

## 3. 功能性需求

### 3.1 需求理解模块 (FR-001 ~ FR-004)
| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-001 | 支持中文自然语言需求描述输入 | 高 |
| FR-002 | 提取并发需求特征（低/中/高/极高） | 高 |
| FR-003 | 提取一致性约束（最终一致/强一致/事务等） | 高 |
| FR-004 | 识别数据流特征（流程型/事件型/CRUD型等） | 中 |

### 3.2 架构推荐模块 (FR-005 ~ FR-010)
| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-005 | 知识库覆盖≥10种架构风格（分层、微服务、事件驱动、CQRS、管道-过滤器、SOA、六边形、MVC、Space-Based、P2P、Serverless、插件架构） | 高 |
| FR-006 | 返回≥3种候选架构，按匹配度降序排列 | 高 |
| FR-007 | 规则引擎+LLM混合推理，规则引擎进行关键词匹配加分，LLM进行语义匹配 | 高 |
| FR-008 | 支持规则引擎反偏见与课程经典风格补入，防止LLM偏好流行架构而忽略最适方案 | 高 |
| FR-009 | 输出匹配理由（含正向加分说明和排除原因） | 中 |
| FR-010 | 支持架构模式组合推荐（如微服务+CQRS+事件驱动） | 中 |

### 3.3 决策支持模块 (FR-011 ~ FR-014)
| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-011 | 多维度对比矩阵（6维度 × N候选架构） | 高 |
| FR-012 | 架构拓扑图可视化（根据用户需求和首选架构生成定制拓扑，固定SVG作为fallback） | 高 |
| FR-013 | 决策溯源（展示从需求特征到架构选择的因果链） | 高 |
| FR-014 | 生成专业化评估报告（含推荐理由、优缺点、风险管理建议） | 高 |

### 3.4 知识进化模块 (FR-015 ~ FR-017)
| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-015 | 每次分析自动保存案例（prompt + features + candidates + report） | 高 |
| FR-016 | Few-shot上下文注入：检索相似历史案例作为匹配Agent的参考 | 高 |
| FR-017 | 案例去重：相同prompt累加计数而非重复存储 | 中 |

### 3.5 Web 交互模块 (FR-018 ~ FR-022)
| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-018 | 深色主题自适应Web UI | 高 |
| FR-019 | 语音输入（Web Speech API，支持中文语音识别） | 中 |
| FR-020 | 语音播报推荐结果（TTS） | 低 |
| FR-021 | Markdown格式渲染（表格、代码块、列表等） | 中 |
| FR-022 | 响应式布局（桌面端网格，移动端单列） | 中 |

## 4. 非功能性需求

### 4.1 性能
| ID | 需求描述 | 指标 |
|----|----------|------|
| NFR-001 | 单次分析响应时间 | ≤ 180s（含LLM调用） |
| NFR-002 | LLM结果缓存 | TTL 300s，命中时秒级返回 |
| NFR-003 | 并发分析能力 | 支持3-5并发请求 |

### 4.2 可用性
| ID | 需求描述 | 指标 |
|----|----------|------|
| NFR-004 | 服务健康检查 | 所有微服务提供 /health 端点 |
| NFR-005 | 优雅降级 | LLM润色失败不阻断主流程 |

### 4.3 AI系统特有需求 — 不确定性处理
| ID | 场景 | 处理方案 |
|----|------|----------|
| NFR-006 | LLM输出格式不稳定 | 规则引擎进行结果约束和校验（score clamping、候选数保证≥3） |
| NFR-007 | 需求描述模糊性 | Few-shot Prompt Engineering（案例库提供上下文示例） |
| NFR-008 | 混合推理冲突 | 规则引擎优先级 > LLM推理；规则引擎加分不抹除LLM匹配 |
| NFR-009 | 模型幻觉防控 | 规则引擎覆盖21种架构风格，防止主流架构偏好 |

### 4.4 可维护性 & 可扩展性
| ID | 需求描述 |
|----|----------|
| NFR-010 | 新增架构风格仅需修改知识库JSON文件 + graph.py中匹配规则 |
| NFR-011 | 微服务独立部署，单服务升级不影响其他模块 |
| NFR-012 | 配置文件驱动（.env环境变量、config.yaml） |

## 5. 约束条件

### 5.1 技术约束
- 采用微服务架构实现系统主体
- 至少3类智能体（需求解析Agent、架构匹配Agent、评估生成Agent）
- 集成大语言模型作为底层支持（DeepSeek）
- 本地开发环境优先使用 Windows PowerShell 启动脚本，也保留 Docker Compose 配置
- 前端为 Vue 3 + Vite 单页应用，构建后由 API Gateway 托管

### 5.2 环境约束
- 操作系统：Windows 本地演示环境优先；WSL/Docker 可作为备选环境
- Python 3.12+
- 无需GPU
- 通过HTTP通信的微服务集群

### 5.3 交付约束
- 开发文档：需求规格说明书、架构设计文档、测试报告
- 验收文档：答辩材料、验收小组分工文档、PR 草稿
- 可运行演示系统 + Web API接口
- 测试数据集 ≥29个场景
- 答辩材料（5min演示 + 15min讲解PPT + 5min问答）

| 编号 | 验收项 | 通过条件 |

| AC-004 | 知识库覆盖 | 覆盖≥10种体系结构风格 |
| AC-005 | 可视化 | UI展示候选卡片、质量属性雷达图、需求定制拓扑图、决策溯源和组合推荐 |

| AC-007 | 知识进化 | 案例库自动增长，Few-shot上下文可注入 |
| AC-008 | 缓存 | 相同需求二次请求秒级返回 |
| AC-009 | 文档交付 | 需求规格说明书+架构设计+测试报告完整 |
| AC-010 | 答辩交付 | 演示脚本+PPT+问答准备完毕 |
