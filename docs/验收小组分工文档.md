# Architecture Assistant 项目验收小组分工文档

项目名称：基于大模型的软件架构风格智能助手  
适用阶段：课程项目最终验收、现场演示、答辩准备、实验报告整理  
当前判断：项目主体功能、前端展示和规则效果已经基本可用于验收，后续重点不是继续大改推荐规则，而是把验收材料、现场脚本、代码注释和讲解链路补齐。

## 一、验收目标与总体策略

本次验收需要证明三件事：

1. 项目能完整运行：从前端输入需求，到后端多服务调用，再到架构推荐、可视化展示、报告生成，链路完整。
2. 项目讲得清楚：PPT、源代码注释、实验报告能对应到真实代码，不停留在概念层面。
3. 项目经得起现场提问：每个核心模块有人负责讲解，有现场可运行的测试脚本，有 Neo4j/知识库增强内容，有进一步优化说明。

验收准备优先级：

| 优先级 | 工作 | 原因 |
|---|---|---|
| P0 | PPT | 最重要，老师主要通过 PPT 和演示判断项目完整度、技术深度和分工情况 |
| P0 | 现场演示和测试脚本 | 防止验收现场只讲不跑，必须能快速证明系统可用 |
| P1 | 实验报告 | 用于补充设计、实现、测试、问题分析和分工记录 |
| P1 | 源代码注释 | 老师追问代码时能快速定位，并体现实现是自己理解过的 |
| P1 | Neo4j 完善 | 作为项目亮点，说明知识图谱增强和可扩展知识库 |
| P2 | 讲解稿和代码优化 | 用于应对问答，修掉明显小问题，提升整体观感 |

建议按照 5 个方向分工，每个方向指定 1 名主负责人，其他成员交叉支持。若组员人数少，可以合并“Neo4j 完善”和“源码注释”，但 PPT 负责人、测试负责人、讲解负责人不建议合并。

| 角色 | 建议负责人 | 主要任务 | 最终交付物 |
|---|---|---|---|
| A. PPT 与答辩统筹 | 组员 A | 制作详细 PPT，串联项目背景、架构、代码、演示和总结 | `项目验收PPT.pptx`、PPT 讲稿 |
| B. 测试与现场脚本 | 组员 B | 准备可现场运行的启动、健康检查、接口测试、批量回归脚本 | 测试文档、测试脚本、测试结果截图 |
| C. Neo4j 与知识库完善 | 组员 C | 完善 Neo4j 知识图谱说明、初始化脚本、查询展示和 fallback 逻辑说明 | Neo4j 说明页、截图、初始化验证记录 |
| D. 源代码注释与实验报告 | 组员 D | 给核心代码补充注释，整理实验报告中的实现细节 | 注释后的源码、实验报告主体 |

## 三、A 组员：PPT 与答辩统筹

### 1. 工作定位

PPT 是本次验收最重要的材料。它不能只写“用了大模型、用了微服务、用了知识图谱”，必须对照项目具体代码，把“项目做了什么、怎么实现、代码在哪里、验收怎么演示”讲清楚。

### 2. PPT 建议结构

建议 18 到 22 页，答辩时控制在 8 到 12 分钟。

| 页码 | 标题 | 内容重点 | 对应代码或材料 |
|---|---|---|---|
| 1 | 项目封面 | 项目名称、课程名称、小组成员、日期 | 无 |
| 2 | 项目背景 | 架构选型依赖经验，需求自然语言不结构化，LLM 需要规则约束 | `README.md`、`docs/架构设计文档.md` |
| 3 | 课程要求对应 | 微服务、多 Agent、LLM、架构知识库、测试场景、Web 演示 | `README.md` 作业要求对应表 |
| 4 | 系统整体架构 | Browser/Vue -> API Gateway -> Orchestration -> Agent Runtime -> LLM Router | `apps/*/main.py` |
| 5 | 微服务划分 | 4 个服务的端口、职责、接口 | `api_gateway/main.py`、`orchestration_engine/main.py`、`agent_runtime/main.py`、`llm_router/main.py` |
| 6 | 前端交互流程 | 输入需求、SSE 流式返回、候选卡片、雷达图、拓扑图、报告 | `frontend/src/App.vue`、`useSSE.ts` |
| 7 | 多 Agent 流程 | 需求解析 Agent、架构匹配 Agent、评估 Agent 串联 | `apps/agent-runtime/agent_runtime/graph.py` |

| 14 | Neo4j 知识图谱 | ArchitectureStyle、Characteristic、UseCase、Keyword 节点和关系 | `init_neo4j.py`、`neo4j_kb.py` |
| 15 | RAG/知识库 | JSON 知识库、RAG 索引、相似案例学习 | `data/architecture_styles.json`、`rag.py`、`build_rag_index.py` |
| 16 | 测试设计 | 20/29 个场景、健康检查、接口测试、端到端演示 | `run_tests.py`、`data/test_scenarios.json` |
| 17 | 测试结果 | 准确率、通过率、失败原因、已知问题 | `test_results.json`、`docs/测试报告.md` |
| 18 | 现场演示路线 | 银行核心交易系统或即时通讯系统案例 | `docs/答辩材料.md` |

| 21 | 小组分工 | 每个人负责内容、代码模块和交付物 | 本文档 |

### 3. PPT 必须写细的地方

PPT 中最容易被老师追问的地方，要写成“概念 + 代码 + 运行效果”三段式。

示例 1：微服务不是口头说说。

- 概念：系统拆分为 API Gateway、Orchestration Engine、Agent Runtime、LLM Router。
- 代码：四个服务分别位于 `apps/api-gateway/`、`apps/orchestration-engine/`、`apps/agent-runtime/`、`apps/llm-router/`。
- 运行效果：启动后访问 `3000/health`、`8001/health`、`8002/health`、`8003/health`，均返回 healthy。

示例 2：多 Agent 不是一个 prompt。

- 概念：需求解析、架构匹配、评估报告生成由不同 Agent 分阶段完成。
- 代码：核心逻辑集中在 `apps/agent-runtime/agent_runtime/graph.py`。
- 运行效果：前端先展示特征，再展示候选架构，最后展示评估报告，说明流程是分阶段返回的。

示例 3：前端展示不是静态页面。

- 概念：前端通过 SSE 获取后端流式结果。
- 代码：`frontend/src/App.vue` 管理整体状态，`useSSE.ts` 负责流式连接，`CandidateCards.vue`、`RadarChart.vue`、`TopologyDiagram.vue` 分别负责候选展示、质量属性图、拓扑图。
- 运行效果：点击“开始分析”后，页面逐步出现特征、候选卡片、图表和报告。

### 4. PPT 负责人交付标准

1. PPT 每页都有标题、图示或表格，避免大段文字。
2. 涉及实现的页面必须标注代码路径。
3. 至少包含一张系统架构图、一张服务调用图、一张 Agent 流程图、一张知识图谱结构图。
4. 演示页必须写清楚现场输入案例、预期结果和出现问题时的备用方案。
5. 结束页必须包含小组分工和项目不足，体现真实工程过程。

## 四、B 组员：测试、测试文档和现场脚本

### 1. 工作定位

测试负责人要保证验收现场不是“只看 PPT”，而是能直接运行项目、跑健康检查、跑接口测试、跑批量场景。现场测试脚本必须简单、稳定、可解释。

### 2. 需要准备的测试内容

| 测试类型 | 目的 | 推荐材料 |
|---|---|---|
| 启动测试 | 证明所有服务能启动 | `scripts/start-local-windows.ps1` |
| 健康检查 | 证明四个服务都在线 | `GET /health` |
| 单接口测试 | 证明后端推荐链路可用 | `POST /api/v1/analyze` |

### 3. 现场测试命令建议

启动服务：

```powershell
powershell -ExecutionPolicy Bypass -File scripts\start-local-windows.ps1
```

健康检查：

```powershell
Invoke-WebRequest -UseBasicParsing http://127.0.0.1:3000/health
Invoke-WebRequest -UseBasicParsing http://127.0.0.1:8001/health
Invoke-WebRequest -UseBasicParsing http://127.0.0.1:8002/health
Invoke-WebRequest -UseBasicParsing http://127.0.0.1:8003/health
```

普通分析接口：

```powershell
Invoke-RestMethod `
  -Method Post `
  -Uri http://127.0.0.1:3000/api/v1/analyze `
  -ContentType "application/json" `
  -Body '{"prompt":"开发银行核心交易系统，需要处理转账、存款、贷款等业务，对数据一致性和审计追踪有极高要求，支持日均百万笔交易。","session_id":"acceptance-demo"}'
```

```powershell

```

### 4. 测试文档要写的内容

测试文档可以基于 `docs/测试报告.md` 扩写，建议补充：

1. 测试环境：Windows、Python 版本、Node 版本、浏览器版本、端口。
2. 启动步骤：如何安装依赖、如何启动四个服务、如何打开前端。
3. 测试用例表：编号、输入需求、预期推荐、实际推荐、是否通过。
4. 现场演示用例：至少准备银行交易系统、即时通讯系统、电商系统三个。
5. 异常处理：LLM Key 缺失、Neo4j 未启动、某个端口被占用、接口超时的处理方式。
6. 测试结论：通过率、剩余问题、改进方向。

### 5. 测试负责人交付标准

1. 能一键启动项目，并知道各服务端口。
2. 能现场跑至少一个接口测试和一个浏览器演示。
3. 批量测试脚本能输出清晰结果。
4. 测试文档中有截图或终端输出记录。
5. 对失败用例有解释，不把所有问题都归因于“网络波动”。

## 五、C 组员：Neo4j 完善

### 1. 工作定位

Neo4j 是项目亮点，不一定是验收最核心功能，但它能体现项目不是简单 prompt 工具，而是有结构化架构知识库和知识图谱扩展能力。

### 2. 当前 Neo4j 相关代码

| 文件 | 作用 |
|---|---|
| `init_neo4j.py` | 初始化 Neo4j 知识图谱，创建架构风格节点、优缺点、适用场景、关键词和关系 |
| `apps/agent-runtime/agent_runtime/neo4j_kb.py` | 封装 Neo4j 连接、可用性检查、知识库读取和查询 |
| `apps/agent-runtime/agent_runtime/graph.py` | 在推荐流程中使用知识库信息，连接规则引擎和候选生成 |
| `data/architecture_styles.json` | Neo4j 不可用时的 JSON fallback 知识库 |

### 3. 需要完善的内容

| 任务 | 具体要求 |
|---|---|
| 检查初始化脚本 | 确保 `init_neo4j.py` 可运行，能创建 ArchitectureStyle、Characteristic、UseCase、Keyword 等节点 |
| 补充图谱说明 | 在 PPT 或实验报告中画出节点类型和关系类型 |
| 准备截图 | 打开 Neo4j Browser，展示部分节点和关系 |
| 准备查询语句 | 至少准备 3 条 Cypher 查询，用于验收现场展示 |
| 说明 fallback | 如果 Neo4j 未启动，系统可以回退到 JSON 知识库，不影响主流程 |

### 4. Neo4j 展示用 Cypher

查看架构风格总数：

```cypher
MATCH (s:ArchitectureStyle)
RETURN count(s) AS style_count;
```

查看某个架构风格的关键词：

```cypher
MATCH (s:ArchitectureStyle)-[:HAS_KEYWORD]->(k:Keyword)
RETURN s.name, collect(k.name) AS keywords
LIMIT 5;
```

查看架构之间的互补关系：

```cypher
MATCH (a:ArchitectureStyle)-[r:COMPLEMENTS|RELATED_TO]->(b:ArchitectureStyle)
RETURN a.name, type(r) AS relation, b.name, r.reason;
```

### 5. Neo4j 讲解重点

现场可以这样讲：

> 我们把架构风格抽象成 ArchitectureStyle 节点，把优点、缺点、适用场景、关键词抽象成不同节点，并通过 HAS_PRO、HAS_CON、SUITABLE_FOR、HAS_KEYWORD 等关系连接起来。这样做的好处是架构知识不只是写死在 prompt 中，而是可以被查询、扩展和维护。如果 Neo4j 服务不可用，系统也设计了 JSON 知识库 fallback，保证验收现场主流程不受影响。

### 6. Neo4j 负责人交付标准

1. `init_neo4j.py` 能说明清楚，即使现场不运行也知道它做了什么。
2. PPT 中至少有一页 Neo4j 知识图谱结构说明。
3. 实验报告中说明 Neo4j 与 JSON fallback 的关系。
4. 准备 Neo4j Browser 截图或查询结果截图。
5. 能回答“为什么要用 Neo4j，而不是只用 JSON”的问题。

源码注释不是把每一行代码翻译成中文，而是给核心逻辑、接口边界、数据结构和关键分支补充说明。实验报告则要把这些实现整理成正式文档。

1. 每个服务入口顶部说明该服务职责。
2. 每个核心接口说明输入、输出和调用下游。
3. 关键算法处说明“为什么这样写”，例如规则加分、降权、fallback。
4. 前端组件注释说明数据从哪里来、展示什么、什么时候更新。
5. 不要给简单赋值和显而易见的变量写废话注释。

```python
def rule_engine_validate(...):
    ...
```

```ts
// 后端通过 SSE 分阶段返回 features、candidates、report 等事件。
// 前端收到事件后逐步更新页面，避免用户等待完整报告生成后才看到结果。
```


预期讲解：

- 银行交易系统强调强一致性、审计追踪和高并发。

即时通讯系统：

```text
开发一个跨平台即时通讯系统，支持万人同时在线、消息实时推送、可靠投递，并且后续需要扩展音视频通话能力。
```

预期讲解：

- 即时通讯系统强调实时、异步、高并发和可靠投递。

### 4. 代码讲解重点

讲解时不要从项目目录一行一行读，而是按调用链讲：

```text
前端 App.vue
  -> API Gateway /api/v1/analyze 或 /api/v1/analyze/stream
  -> Orchestration Engine 编排请求
  -> Agent Runtime 运行多 Agent 流程
  -> graph.py 提取特征、匹配架构、规则校验、生成报告
  -> LLM Router 调用 DeepSeek
  -> 前端展示候选、图表、报告
```

重点文件：

| 文件 | 讲解角度 |
|---|---|
| `frontend/src/App.vue` | 页面状态如何随 SSE 事件变化 |
| `apps/api-gateway/api_gateway/main.py` | 为什么需要统一入口 |
| `apps/orchestration-engine/orchestration_engine/main.py` | 为什么需要编排层和缓存 |
| `apps/agent-runtime/agent_runtime/graph.py` | 三个 Agent 和规则引擎是核心 |
| `apps/llm-router/llm_router/main.py` | 为什么不让前端直接调大模型 |
| `init_neo4j.py` | 知识图谱如何初始化 |

## 十、老师可能追问与回答方向

### 1. 你们为什么要拆成多个服务？

回答方向：

> 因为项目本身要体现软件体系结构设计。我们把前端入口、流程编排、Agent 推理、大模型调用拆开，API Gateway 负责统一入口，Orchestration Engine 负责编排和缓存，Agent Runtime 负责核心推荐逻辑，LLM Router 负责模型调用代理。这样职责清晰，也方便后续替换模型或扩展 Agent。

### 2. 你们的大模型是不是只是在生成一段文字？

回答方向：

> 不是。大模型主要用于语义理解和报告润色，推荐结果还会经过知识库匹配和规则引擎校验。核心规则在 `graph.py`，前端展示的候选架构、匹配理由、风险提示和报告是经过多阶段处理后返回的。

### 3. 规则引擎的作用是什么？

回答方向：

> 规则引擎用于增强稳定性和可解释性。LLM 结果可能波动，规则引擎会根据并发、一致性、审计、实时性、模块化等关键词对候选架构加权或降权，使推荐结果更符合软件架构知识和课程场景。

### 4. Neo4j 在项目中起什么作用？

回答方向：

> Neo4j 用来表示架构风格知识图谱。架构风格、优点、缺点、适用场景、关键词都可以作为节点和关系存储。这样架构知识可以被查询和扩展。如果验收现场 Neo4j 没启动，系统会回退到 JSON 知识库，保证主流程正常。

### 5. 你们如何证明项目可运行？

回答方向：

> 我们准备了四个服务的健康检查、接口测试、批量测试脚本和前端端到端演示。现场可以访问 `http://127.0.0.1:3000/` 输入需求，也可以用 PowerShell 调接口验证结果。

### 6. 项目还有什么不足？

回答方向：

> 目前不足主要有三个：第一，推荐规则还可以用更多真实案例继续调优；第二，Neo4j 目前主要作为知识库增强，后续可以进一步参与推荐排序；第三，测试脚本需要继续增强跨平台兼容和自动化报告生成。
