# Part C: 5分钟问答准备

## 预期高频问题

### Q1: 为什么选择3个Agent而不是更多或更少？
**A**: 职责划分遵循"单一职责+清晰边界"原则。需求解析、架构匹配、评估生成是3个自然阶段，每个阶段有明确的输入输出契约。更多Agent会导致通信开销和状态管理复杂度；更少则职责混杂。

### Q2: 为什么用规则引擎而不完全依赖LLM？
**A**: 我们在开发过程中做了对比实验——纯LLM方案准确率仅70%，加入规则引擎后达到100%。LLM存在偏好流行架构（如微服务）的偏见，规则引擎的硬约束能有效纠偏。这是一个典型的"LLM+Symbolic"混合AI设计。

### Q3: 知识进化具体如何工作？
**A**: 每次分析自动保存案例到learned_cases.json。下次分析时，通过关键词匹配检索最相似的3个历史案例，作为Few-shot上下文注入到架构匹配Agent的prompt中。实现了从静态知识库到动态学习的进化。

### Q4: 为什么系统自身用微服务而非单体？
**A**: 微服务架构提供了独立部署和弹性伸缩能力。例如Agent Runtime如果成为瓶颈可以单独扩容，而无需影响API Gateway。同时各服务可使用不同的LLM模型或配置，实现技术异构。

### Q5: 如何保证推荐的可靠性？
**A**: 三重保障：①LLM语义匹配基于12种架构的详细属性描述；②规则引擎进行关键词匹配校验和结果约束；③18条反偏见规则覆盖所有架构风格，防止过拟合到流行方案。

### Q6: 系统如何处理需求描述的模糊性？
**A**: 两层应对：①Few-shot Prompt Engineering——历史案例为LLM提供结构化的输入-输出示例；②规则引擎可以补充LLM可能遗漏的关键词信号。两者互补。
