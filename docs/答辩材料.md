# 答辩材料

项目名称：基于大模型的软件架构风格智能助手  
英文名称：Architecture Assistant  
建议答辩结构：5 分钟系统演示 + 15 分钟架构设计讲解 + 5 分钟问答

## 一、答辩目标

本次答辩需要证明三件事：

1. 系统满足《软件体系结构》课程大作业要求。
2. 架构设计合理，体现微服务、多 Agent、LLM 和规则引擎的工程化集成。
3. 系统可运行、可解释、可演示，能够从需求输入完整走到架构推荐与评估报告。

### 2. 页面状态

打开：

```text
http://localhost:3000/
```

确认：

- 页面标题为 Architecture Assistant。
- 输入框、示例场景、日夜模式按钮可见。
- 点击示例场景可以填入需求。
- 点击“开始分析”后，特征会先出现，候选架构随后出现，报告最后打字机显示。

### 3. 推荐演示输入

```text
开发银行核心交易系统，需要处理转账、存款、贷款等业务，对数据一致性和审计追踪有极高要求，支持日均百万笔交易。
```

推荐这个案例的原因：

- 触发金融、交易、强一致、审计、高并发等清晰架构信号。
- 通常推荐 CQRS，易于解释。
- 可以展示候选架构、拓扑图、决策溯源和报告。

## 三、5 分钟系统演示脚本

### 0:00 - 0:30 开场

讲稿：

> 各位老师好，我们展示的是“基于大模型的软件架构风格智能助手”。这个系统面向软件架构选型场景，可以从自然语言需求出发，自动完成需求理解、候选架构推荐、多维度评估和报告生成。

操作：

1. 打开 `http://localhost:3000/`。
2. 指出页面顶部的 4 个微服务和 3 个智能体标识。

要点：

- 系统主体采用微服务架构。
- 内部由 3 个 Agent 协作完成分析。
- 前端用于交互和可视化展示。

### 0:30 - 1:20 输入需求并启动分析

讲稿：

> 用户可以输入自然语言需求，也可以使用语音输入。这里我们以银行核心交易系统为例，它包含转账、存款、贷款、强一致性和审计追踪等典型架构约束。

操作：

1. 点击银行核心交易系统示例。
2. 点击“开始分析”。

观察：

- 页面显示“正在连接分析服务”。
- 随后先出现 Feature Extraction。

讲稿：

> 系统首先进入需求解析阶段，需求解析 Agent 会提取并发、一致性、事务、审计追踪等关键特征。这里可以看到特征不是等最终报告完成才出现，而是通过 SSE 流式返回逐步展示。

### 1:20 - 2:20 展示候选架构推荐

观察：

- 候选卡片出现。
- 首选一般为 CQRS。
- 备选通常包括六边形架构、微服务或分层架构。

讲稿：

> 第二阶段是架构匹配。架构匹配 Agent 会结合知识库、历史案例和规则引擎返回至少 3 种候选架构。每个候选架构都有匹配度、推荐理由和风险提示。

重点解释：

- CQRS 适合读写负载差异大、强审计、强一致场景。
- 六边形架构适合保护复杂核心领域逻辑。
- 微服务适合多模块独立部署和扩展。

### 2:20 - 3:30 展示可视化与决策溯源

操作：

滚动到质量属性雷达、拓扑图、决策溯源区域。

讲稿：

> 系统不仅给出结论，还提供决策支持。质量属性雷达用于比较候选架构在扩展性、性能、耦合度、复杂度、部署弹性和可测试性上的差异。

> 拓扑图用于展示架构结构。当前 CQRS 图展示了命令处理层、查询服务层、写库、读库和事件投影之间的关系。

> 决策溯源用于解释为什么推荐这个架构。例如“转账、存款、贷款”等核心交易需求触发 CQRS，审计追踪需求可以通过事件溯源满足，高并发读写场景可以通过读写分离优化。

### 3:30 - 4:30 展示报告与组合推荐

操作：

滚动到 Combination 和 Evaluation Report。

讲稿：

> 系统还会给出组合推荐，例如以 CQRS 作为核心架构，同时搭配六边形架构保护领域模型。最终报告会从多维度对比、架构影响溯源、优缺点和风险管理建议等方面形成结构化分析。

强调：

- 报告使用打字机效果逐步显示。
- Markdown 表格、标题和列表已正确渲染。

### 4:30 - 5:00 总结

讲稿：

> 总结来说，本系统完成了从需求输入、特征提取、架构推荐、可视化分析到报告生成的完整流程。它不是单纯调用大模型，而是采用 LLM + 知识库 + 规则引擎 + 多 Agent 的复合 AI 系统设计，提升了推荐结果的可解释性和可靠性。

### 第 2 页：问题背景

## 五、问答准备

### Q1：为什么系统自身要用微服务架构？

答：

> 作业要求建议采用微服务架构。我们的系统有天然的服务边界：前端入口、编排服务、Agent 推理服务、LLM 代理服务。拆分后每个服务职责清晰，Agent Runtime 和 LLM Router 也可以独立扩展。

### Q2：为什么需要 3 个 Agent？

答：

> 架构推荐过程可以自然拆成三个阶段：需求解析、架构匹配、评估生成。每个 Agent 有独立输入输出，既符合单一职责原则，也方便调试和扩展。

### Q3：为什么不完全依赖大模型？

答：

> 纯 LLM 输出不稳定，而且容易偏向微服务、事件驱动等热门架构。我们加入规则引擎，对强信号进行加分或降权，使结果更可控、更可解释。

### Q4：知识库为什么先用 JSON，而不是完全使用 Neo4j？

答：

> Neo4j 是作业建议项，不是强制项。项目已经预留 Neo4j 接入能力，但为了保证答辩演示稳定，默认使用结构化 JSON 知识库。Neo4j 不可用时系统会自动回退，不影响核心功能。

### Q5：系统如何处理需求模糊性？

答：

> 第一，需求解析 Agent 会把自然语言转成结构化特征。第二，架构匹配 Agent 会结合历史案例进行 Few-shot 推理。第三，规则引擎会对明显架构信号做校验。

### Q6：如何证明推荐结果可解释？

答：

> 前端展示推荐理由、风险提示、质量属性雷达、架构拓扑图和决策溯源。报告中也包含多维度对比和最终推荐依据。

### Q7：系统有哪些异常处理？

答：

> Neo4j 不可用会回退 JSON，LLM 润色失败不阻断主流程，SSE 失败前端会显示错误或 fallback 到普通分析，服务均提供 health check。

### Q8：如果要继续完善，优先做什么？

答：

> 优先引入向量检索和 Neo4j 完整知识图谱，其次做多轮需求澄清，让系统在需求不足时主动提问，而不是直接给出一次性推荐。
