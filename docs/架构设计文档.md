# 架构设计文档

## 1. 项目概述

**项目名称**：基于大模型的软件架构风格智能助手  
**英文名称**：Architecture Assistant  
**项目定位**：面向软件体系结构课程大作业的智能架构决策支持系统。

系统接收用户自然语言描述的软件需求，通过大语言模型、多智能体协作和规则引擎混合推理，自动推荐候选软件体系结构风格，并生成可解释的评估报告和可视化结果。

## 2. 设计目标

1. 支持自然语言需求输入，提取关键功能与非功能需求。
2. 推荐至少 3 种候选体系结构风格。
3. 输出最终推荐、匹配理由、风险提示和优缺点评估。
4. 提供对比矩阵、质量属性可视化、拓扑图和决策溯源。
5. 支持知识库扩展与案例学习。
6. 采用微服务架构和至少 3 类 Agent 实现系统主体。

## 3. 总体架构

系统自身采用微服务架构。整体由 4 个后端服务、1 个 Vue 前端和 2 类数据文件组成。

```text
┌──────────────────────────────────────────────┐
│ Browser / Vue Frontend                        │
│ 输入需求、语音交互、主题切换、结果可视化       │
└───────────────────────┬──────────────────────┘
                        │ HTTP / SSE
┌───────────────────────▼──────────────────────┐
│ API Gateway :3000                             │
│ Web 静态资源托管、REST/SSE 转发、统一入口       │
└───────────────────────┬──────────────────────┘
                        │ HTTP / SSE
┌───────────────────────▼──────────────────────┐
│ Orchestration Engine :8001                    │
│ 分析编排、缓存管理、Agent Runtime 调度          │
└───────────────┬───────────────────┬──────────┘
                │                   │
                │ HTTP / SSE         │ HTTP
┌───────────────▼──────────────┐    │
│ Agent Runtime :8003           │    │
│ LangGraph 三 Agent 协作        │    │
│ 规则引擎、知识进化、案例学习    │    │
└───────────────┬──────────────┘    │
                │                   │
                │ LLM 调用           │
┌───────────────▼───────────────────▼──────────┐
│ LLM Router :8002                              │
│ DeepSeek API 代理、模型调用统一入口             │
└──────────────────────────────────────────────┘
```

## 4. 微服务划分

### 4.1 API Gateway

| 属性 | 内容 |
|---|---|
| 端口 | 3000 |
| 框架 | FastAPI |
| 主要职责 | 统一入口、Vue 静态资源托管、REST/SSE 转发 |
| 关键接口 | `GET /`、`GET /health`、`POST /api/v1/analyze`、`POST /api/v1/analyze/stream` |
| 依赖 | Orchestration Engine |

设计说明：

- API Gateway 负责托管 `frontend/dist`，浏览器访问 `http://localhost:3000/` 即可打开新版 Vue 前端。
- `/api/v1/analyze/stream` 以 SSE 方式转发编排服务返回的阶段性结果。
- 该层不直接接触 LLM API Key。

### 4.2 Orchestration Engine

| 属性 | 内容 |
|---|---|
| 端口 | 8001 |
| 框架 | FastAPI + httpx |
| 主要职责 | 分析流程编排、缓存、上游服务协调 |
| 关键接口 | `POST /api/v1/analyze`、`POST /api/v1/analyze/stream`、`GET /health` |
| 依赖 | Agent Runtime、LLM Router |

分析流程：

```text
用户需求
  -> 缓存检查
  -> Agent Runtime 分析
  -> 候选架构与报告返回
  -> 写入缓存
  -> 返回给 API Gateway
```

流式流程：

```text
API Gateway
  -> Orchestration Engine /api/v1/analyze/stream
  -> Agent Runtime /api/v1/run/stream
  -> features event
  -> candidates event
  -> report event
  -> done event
```

### 4.3 Agent Runtime

| 属性 | 内容 |
|---|---|
| 端口 | 8003 |
| 框架 | FastAPI + LangGraph |
| 主要职责 | 三 Agent 协作、知识库匹配、规则引擎、案例学习 |
| 关键接口 | `POST /api/v1/run`、`POST /api/v1/run/stream`、`GET /api/v1/knowledge`、`GET /api/v1/cases` |
| 依赖 | DeepSeek/LLM Router、JSON 知识库、可选 Neo4j |

Agent Runtime 是系统核心，内部包含：

- `RequirementAnalysisAgent`：需求解析与特征提取。
- `ArchitectureMatchingAgent`：知识库匹配与候选架构排序。
- `EvaluationAgent`：评估报告生成。
- `rule_engine_validate`：规则引擎校验、加分、降权。
- 知识进化逻辑：历史案例保存、去重、Few-shot 注入和相似案例命中展示。

### 4.4 LLM Router

| 属性 | 内容 |
|---|---|
| 端口 | 8002 |
| 框架 | FastAPI + httpx |
| 主要职责 | 统一 LLM 请求入口，代理 DeepSeek API |
| 关键接口 | `POST /api/v1/generate`、`GET /health` |

设计理由：

- 避免前端直接暴露 API Key。
- 后续可平滑切换模型供应商。
- 可统一处理超时、重试、模型参数和日志。

## 5. 前端架构

当前前端位于 `frontend/`，采用 Vue 3 + Vite 构建，由 API Gateway 托管构建后的 `frontend/dist`。

主要组件：

| 组件 | 职责 |
|---|---|
| `App.vue` | 页面状态管理、SSE 流式消费、主题切换、打字机报告 |
| `InputPanel.vue` | 文本输入、示例场景、语音输入和语音通话入口 |
| `CandidateCards.vue` | 候选架构卡片、匹配度、理由和风险展示 |
| `FeatureTags.vue` | 需求特征标签 |
| `RadarChart.vue` | 多维质量属性雷达图 |
| `TopologyDiagram.vue` | 需求定制拓扑图渲染，兼容固定 SVG fallback |
| `DecisionTrace.vue` | 决策溯源因果链 |
| `ComboRec.vue` | 架构组合推荐 |
| `ReportPanel.vue` | Markdown 报告渲染 |

用户体验增强：

- SSE 分阶段结果展示，避免等待完整响应。
- 报告正文打字机效果。
- 日夜主题切换。
- 半透明磨砂面板与多色视觉层次。
- 语音输入和连续通话入口。

## 6. 三 Agent 协作机制

### 6.1 状态流转

```text
AgentState
  user_requirement
  extracted_features
  candidate_styles
  evaluation_report
  case_context
```

LangGraph 将三个 Agent 串联为确定性流水线：

```text
RequirementAnalysisAgent
        |
        v
ArchitectureMatchingAgent
        |
        v
EvaluationAgent
        |
        v
END
```

### 6.2 需求解析 Agent

输入：自然语言需求。  
输出：结构化特征。

提取内容包括：

- 应用领域
- 并发量
- 一致性要求
- 可扩展性需求
- 部署灵活性
- 关键业务需求
- 数据流特征

### 6.3 架构匹配 Agent

输入：

- `extracted_features`
- 架构风格知识库摘要
- 相似历史案例 `case_context`

输出：

- 至少 3 个候选架构
- 匹配分数
- 推荐理由
- 风险提示

匹配机制：

1. LLM 进行语义匹配。
2. 知识库提供架构风格属性、适用场景和排除场景。
3. 规则引擎对候选结果进行加分、降权和排序。
4. 返回结构化 JSON。

### 6.4 评估生成 Agent

输入：

- 用户需求
- 候选架构
- 匹配理由

输出：

- 多维度对比分析
- 最终推荐
- 优缺点总结
- 风险管理建议
- 架构组合建议

## 7. 混合推理设计

系统不完全依赖 LLM，而采用 **LLM + 规则引擎** 的混合推理模式。

### 7.1 LLM 的职责

- 理解自然语言中的隐含需求。
- 根据知识库描述进行语义匹配。
- 生成专业自然语言评估报告。

### 7.2 规则引擎的职责

- 对强信号关键词进行加分。
- 对明显不适用的架构进行降权。
- 抑制 LLM 对微服务、事件驱动等流行架构的惯性偏好。
- 保证候选结果数量、分数范围和排序稳定性。

典型规则：

| 需求信号 | 推荐倾向 |
|---|---|
| 银行、交易、转账、强一致、审计 | CQRS |
| 视频转码、流水线、ETL、日志处理 | 管道-过滤器 |
| 插件、IDE、网关中间件链 | 插件/微内核 |
| 去中心化、P2P、区块链、CDN | 对等架构 |
| 多业务模块、独立部署、多团队 | 微服务 |
| 外部系统适配、复杂领域模型 | 六边形架构 |

## 8. 知识库设计

### 8.1 架构风格知识库

文件：`data/architecture_styles.json`

当前覆盖 21 种架构风格，既包含课程经典五大类，也保留现代演进风格：

1. 分层架构
2. 微服务架构
3. 事件驱动架构
4. CQRS
5. 管道-过滤器
6. SOA
7. 六边形架构/端口适配器
8. MVC
9. Space-Based
10. 对等架构
11. Serverless
12. 插件/微内核架构
13. 批处理架构
14. 主程序-子过程架构
15. 面向对象架构
16. 仓库架构
17. 黑板架构
18. 解释器架构
19. 规则系统架构
20. 进程通信架构
21. 多 Agent 架构

每个条目包含：

- 名称与别名
- 分类
- 描述
- 质量属性
- 适合场景
- 不适合场景
- 优点与缺点
- 关键技术
- 典型案例

### 8.2 Neo4j 可选增强

作业建议使用 Neo4j。当前项目已预留 Neo4j 查询能力：

- Neo4j 可用时优先读取图数据库。
- Neo4j 不可用时自动回退到 JSON 知识库。
- `init_neo4j.py` 可初始化 ArchitectureStyle、Characteristic、UseCase、Keyword 节点，以及 COMPLEMENTS、RELATED_TO 等架构关系。

这样可以保证演示环境稳定，同时保留知识图谱扩展路径。

### 8.3 案例学习库

文件：`data/learned_cases.json`

机制：

1. 每次分析后保存案例。
2. 相同 prompt 去重并累加计数。
3. 后续请求检索相似案例。
4. 将相似案例注入架构匹配 Agent 的 prompt，形成 Few-shot 上下文。

当前仓库保留 36 条历史案例，前端决策溯源区域会展示相似案例命中，帮助解释推荐依据。

## 9. 数据流设计

### 9.1 普通分析接口

```text
POST /api/v1/analyze
```

返回完整 JSON：

```json
{
  "session_id": "demo",
  "features": {},
  "candidates": [],
  "report": "...",
  "steps": [],
  "cached": false
}
```

### 9.2 流式分析接口

```text
POST /api/v1/analyze/stream
```

SSE 事件：

```text
data: {"event":"status","message":"..."}

data: {"event":"features","data":{...}}

data: {"event":"case_matches","data":[...]}

data: {"event":"candidates","data":[...]}

data: {"event":"topology","data":{...}}

data: {"event":"report","data":"..."}

data: {"event":"done"}
```

前端收到 `features` 后立即显示特征，收到 `case_matches` 后更新相似案例，收到 `candidates` 后显示候选架构。Agent Runtime 在评估阶段会再次发出经过规则引擎校验后的 `candidates`，确保候选卡片展示的是最终排序；收到 `topology` 后渲染需求定制拓扑图，收到 `report` 后用打字机效果渲染报告。

## 10. 质量属性设计

| 质量属性 | 设计方案 |
---|---|
| 可扩展性 | 微服务划分，LLM Router 与 Agent Runtime 可独立扩容 |
| 可维护性 | 前后端分层，Agent 职责明确，知识库结构化 |
| 可测试性 | 29 个典型场景数据集，健康检查接口，独立服务接口 |
| 可用性 | LLM 润色失败不阻断主流程，Neo4j 不可用时回退 JSON |
| 性能 | 缓存相同 prompt，SSE 分阶段输出降低等待感 |
| 安全性 | API Key 通过环境变量配置，不暴露到前端 |

## 11. 部署设计

### 11.1 本地演示部署

推荐命令：

```powershell
npm.cmd --prefix frontend run build
powershell -ExecutionPolicy Bypass -File scripts\start-local-windows.ps1
```

访问：

```text
http://localhost:3000/
```

### 11.2 Docker 部署

仓库提供 `docker-compose.yml` 和各服务 Dockerfile，可用于容器化部署。实际答辩演示建议使用本地脚本启动，减少 Docker 环境差异带来的风险。

## 12. 异常处理

| 异常场景 | 处理方式 |
|---|---|
| LLM 调用失败 | 重试或返回错误信息 |
| LLM 报告润色失败 | 不阻断主流程，返回原始报告 |
| Neo4j 不可用 | 自动回退 JSON 知识库 |
| SSE 上游断开 | 前端显示错误并尝试普通分析 fallback |
| 语音识别不可用 | 保留文本输入，不影响核心流程 |
