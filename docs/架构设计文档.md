## 1. 架构概述

### 1.1 架构风格选择
本项目**自身采用微服务架构**实现。选择依据：
- **独立部署**：需求理解、架构匹配、评估生成等模块可独立迭代升级
- **技术异构**：各微服务可按需使用不同的模型和依赖
- **弹性伸缩**：LLM推理服务可独立扩容应对高并发
- **渐进交付**：与课程作业迭代开发节奏匹配

### 1.2 系统构成
系统由4个微服务 + 1个前端页面组成：

```
┌─────────────────────────────────────────────────────┐
│                     Browser (UI)                      │
│   语音输入 | Markdown渲染 | 对比矩阵 | 拓扑图         │
└─────────────────────┬───────────────────────────────┘
                      │ HTTP :3000
┌─────────────────────▼───────────────────────────────┐
│              API Gateway (FastAPI :3000)              │
│  路由转发 | 缓存透传 | CORS | HTML渲染              │
└─────────────────────┬───────────────────────────────┘
                      │ HTTP :8001
┌─────────────────────▼───────────────────────────────┐
│         Orchestration Engine (FastAPI :8001)          │
│  流水线编排 | 缓存管理 | LLM润色调度                │
└──────┬─────────────────────────────────┬────────────┘
       │ HTTP :8003                      │ HTTP :8002
┌──────▼──────────────────┐  ┌──────────▼────────────┐
│ Agent Runtime (:8003)   │  │  LLM Router (:8002)   │
│ LangGraph 三Agent协作   │  │  DeepSeek API 代理    │
│ 知识进化 · 案例学习     │  │  统一LLM调用入口      │
└─────────────────────────┘  └───────────────────────┘
```

## 2. 微服务划分

### 2.1 API Gateway（端口3000）
| 属性 | 值 |
|------|-----|
| 职责 | 统一入口，请求路由，Web UI托管 |
| 框架 | FastAPI + uvicorn |
| 依赖 | Orchestration Engine (:8001) |
| 关键接口 | `POST /api/v1/analyze`, `GET /` (HTML) |
| 前端功能 | 语音输入(Web Speech API)、Markdown渲染、对比矩阵(Canvas SVG)、拓扑图(12种)、决策溯源列表 |

**设计决策**：
- 前端内嵌为单页HTML，避免独立前端项目增加复杂度
- 使用HTML5 Web Speech API实现语音输入，无需后端处理音频
- Markdown渲染完全在前端完成，减轻服务端压力

### 2.2 Orchestration Engine（端口8001）
| 属性 | 值 |
|------|-----|
| 职责 | 流水线编排，缓存管理，LLM润色调度 |
| 框架 | FastAPI + httpx(AsyncClient) |
| 依赖 | Agent Runtime (:8003), LLM Router (:8002 可选) |
| 关键接口 | `POST /api/v1/analyze`, `POST /api/v1/analyze/stream` |

**完整流水线**:
```
User Input → Cache检查(命中→直接返回)
           ↓ (未命中)
         Agent Runtime分析（三Agent协作）
           ↓
         LLM 润色报告（可选，失败不阻断）
           ↓
         写入缓存 + 返回结果
```

**缓存设计**：
- 以 `sha256(prompt)` 为key
- TTL: 300秒（可配置）
- 最大条目：500条
- 首次请求完整分析，相同需求二次请求秒级命中

### 2.3 Agent Runtime（端口8003）
| 属性 | 值 |
|------|-----|
| 职责 | 三Agent协作流水线，知识进化 |
| 框架 | FastAPI + LangGraph |
| 依赖 | LLM Router (:8002) |
| 关键接口 | `POST /api/v1/run`, `GET /api/v1/knowledge`, `POST /api/v1/cases` |

### 2.4 LLM Router（端口8002）
| 属性 | 值 |
|------|-----|
| 职责 | LLM API 统一代理，不暴露API Key到前端 |
| 框架 | FastAPI + httpx |
| 关键接口 | `POST /api/v1/generate` |

## 3. 智能体协作机制

### 3.1 三Agent流水线（LangGraph实现）

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│ 需求解析 Agent   │────▶│ 架构匹配 Agent   │────▶│ 评估生成 Agent   │
│ (Feature         │     │ (Architecture   │     │ (Evaluation     │
│  Extraction)     │     │  Matching)      │     │  Generation)    │
└─────────────────┘     └─────────────────┘     └─────────────────┘
        │                       │                       │
   提取特征                 匹配+排序              生成报告
   features,              candidates,             evaluation_report
   constraints            match_scores
```

### 3.2 Agent行为定义

#### Agent 1：需求解析 (Feature Extraction Agent)
```
输入: user_requirement (自然语言)
输出: extracted_features, constraints

Prompt策略：
- 从需求描述中系统化提取10类特征
  1. 领域类型 (电商/金融/物联网/社交/...)
  2. 数据流模式 (请求-响应/事件驱动/流式/批处理)
  3. 并发需求 (低/中/高/极高)
  4. 一致性要求 (最终一致/强一致/事务)
  5. 扩展性需求 (低/中/高)
  6. 响应延迟要求
  7. 模块化水平
  8. 部署约束
  9. 团队规模
  10. 复杂度容忍度
```

#### Agent 2：架构匹配 (Architecture Matching Agent)
```
输入: extracted_features + knowledge_summary + case_context
输出: candidate_styles (≥3个，按match_score排序)

混合推理架构：
┌─────────────────────────────────────────┐
│          Architecture Matching           │
├─────────────────────────────────────────┤
│  1. LLM 语义匹配（DeepSeek API）         │
│     - 18条反偏见规则                     │
│     - Few-shot历史案例                   │
│     - Prompt包含12种架构触发词/排除词     │
├─────────────────────────────────────────┤
│  2. 规则引擎校验 (rule_engine_validate)  │
│     - 8组关键词正向加分                  │
│     - Score clamping [0.0, 1.0]          │
│     - 候选数保证≥3                       │
├─────────────────────────────────────────┤
│  3. 综合排序输出                        │
└─────────────────────────────────────────┘

18条反偏见规则（摘录）：
1. 防止LLM过度偏好微服务
2. 高并发+事件特征 → 事件驱动优先
3. 银行/交易/转账/一致性 → CQRS优先
4. 管道式数据流 → 管道-过滤器优先
5. 去中心化+P2P关键词 → P2P优先
...
18. 无明确架构信号 → 分层架构兜底
```

#### Agent 3：评估生成 (Evaluation Generation Agent)
```
输入: candidates + extracted_features
输出: evaluation_report

报告结构：
1. 评估背景
2. 候选架构得分排行
3. 多维度对比分析（6维度）
4. 架构影响溯源（因果链 ≥3条）
5. 组合建议（微服务+CQRS+事件驱动等）
6. 最终推荐及优缺点
7. 风险管理建议
8. 演进路径建议
```

### 3.3 Few-shot 知识进化

```
分析请求 ─▶ _build_case_context(prompt)
              │
              ▼
        加载 learned_cases.json
              │
              ▼
        关键词匹配排序（取top-3相似案例）
              │
              ▼
        注入 case_context 到 Agent 2 prompt
              │
              ▼
        分析完成后 _save_case() 自动保存
```

## 4. LLM集成方案

### 4.1 模型配置
| 配置项 | 值 |
|--------|-----|
| 模型 | deepseek-v3 (通过SiliconFlow API) |
| API Base | `https://api.siliconflow.cn/v1` |
| 调用方式 | REST API (httpx AsyncClient) |
| 超时 | 120s（Agent Runtime），60s（报告润色） |

### 4.2 Prompt Engineering策略

**三层体系**：
```
Layer 1: System Prompt
  - 角色定义 + 输出格式约束
  - 架构风格知识注入（12种架构触发词/排除词）

Layer 2: Task Prompt
  - 具体需求描述
  - 结构化特征数据

Layer 3: Few-shot Context（知识进化）
  - 3个相似历史案例
  - 每个案例：需求摘要 → 推荐架构
```

### 4.3 可靠性保障
| 机制 | 说明 |
|------|------|
| 重试 | HTTP 5xx错误自动重试3次（0.5s/1.0s/1.5s退避） |
| 格式修复 | JSON解析失败时自动修复常见格式错误 |
| 结果校验 | 规则引擎二次校验所有LLM输出 |
| 优雅降级 | LLM润色失败不阻断主流程 |

## 6. 数据存储

### 6.1 知识库（knowledge_base.json）
```json
[
  {
    "name": "事件驱动架构 (Event-Driven Architecture)",
    "category": "独立构件",
    "scalability": "高",
    "performance": "高（异步处理）",
    "coupling": "低（通过事件解耦）",
    "complexity": "高",
    "deployability": "独立部署",
    "testability": "中（事件流测试复杂）",
    "适合场景": ["高并发", "异步处理", "实时系统"],
    "不适合场景": ["简单CRUD", "强一致性事务"],
    "优点": ["高吞吐量", "松耦合", "扩展性强"],
    "缺点": ["调试困难", "最终一致性", "事件溯源复杂度高"],
    "关键技术": ["Kafka", "RabbitMQ", "Event Sourcing", "CQRS"],
    "典型案例": ["Netflix", "Uber", "LinkedIn"]
  },
  ...
]
```
覆盖12种架构风格。

### 6.2 案例库（learned_cases.json）
```json
[
  {
    "prompt": "...",
    "features": {...},
    "candidates": [...],
    "report": "...",
    "count": 2,
    "created": 1746700000.0,
    "last_used": 1746700300.0
  }
]
```
自动增长，去重累加，为Few-shot提供上下文。

### 6.3 缓存（内存）
- 存储于Orchestration Engine进程内存
- key: sha256(prompt)
- value: {features, candidates, report, timestamp}
- TTL: 300s, Max: 500条

## 7. 关键技术决策

| 决策点 | 方案 | 备选方案 | 决策理由 |
|--------|------|----------|----------|
| 前端框架 | 内嵌HTML | React/Vue独立前端 | 减少依赖，单文件交付，匹配作业要求 |
| 智能体编排 | LangGraph | 手工编排/Chain | 状态管理自动化，流式支持，可观测性 |
| LLM集成 | HTTP REST | LangChain封装 | 减少框架耦合，灵活切换模型 |
| 知识存储 | JSON文件 | Neo4j | 快速构建，无需额外运维；12种风格关系简单 |
| 容器化 | 不采用 | Docker Compose | WSL不可用Docker，本地uvicorn更直观 |
| 规则引擎 | Python函数 | 独立规则引擎 | 规则简单（8组），Python实现即满足 |
| 前端渲染 | 客户端Markdown | 服务端渲染 | 减少服务端计算，即时渲染 |

## 8. 安全与异常处理

### 8.1 API安全
- LLM API Key存储在.env环境变量，不暴露到前端
- LLM Router作为代理层，前端不直接调用LLM API
- CORS严格限制，生产环境需改为仅允许指定Origin

### 8.2 异常处理策略
| 层级 | 策略 |
|------|------|
| LLM调用 | 自动重试3次，超时120s |
| 上游不可用 | 返回502 + 错误详情 |
| LLM润色失败 | 不阻断，返回未润色报告 |
| 案例保存失败 | 不阻断，记录warning日志 |
| 缓存写入失败 | 不阻断，返回本次结果 |
| JSON解析失败 | 返回parse_error，测试统计中记录 |

## 9. 扩展性设计

### 9.1 新增架构风格
1. 在 `knowledge_base.json` 中添加条目
2. 在 `graph.py` 的 `build_knowledge_summary()` 中添加触发词/排除词
3. 在 `rule_engine_validate()` 中添加正向加分规则
4. 在前端HTML的 `buildTopology()` 中添加SVG拓扑图
5. 在前端JS的 `getKB()` 中添加维度评分

### 9.2 新增智能体
1. 在 `graph.py` 中添加新节点函数
2. 在 `StateGraph` 中添加节点和边
3. 在 `AgentState` 中添加新字段

### 9.3 切换到生产模式
1. 使用 Docker Compose 或 Kubernetes 部署
2. 引入 Redis 作为缓存层替代内存缓存
3. 引入消息队列（如RabbitMQ）进行异步分析
4. 数据库（如PostgreSQL）替代JSON文件存储知识库和案例
5. 引入 API Gateway（如Kong/Traefik）替代内嵌FastAPI网关
